# Paper Hub 项目架构与功能文档

## 1. 项目定位

Paper Hub 是一个本地运行的单机论文工作台，目标是把论文管理、PDF 导入、阅读进度、笔记、AI 元数据整理、页面翻译和论文精华提炼放在同一个界面里。

当前实现不是前后端分离部署，也不是多用户系统，而是一个由 Python 进程直接提供静态页面和 JSON API 的本地 Web 应用。

## 2. 总体架构

### 2.1 架构形态

系统采用“单体后端 + 原生前端 + 本地文件存储”的结构：

```text
浏览器
  ├─ index.html
  ├─ app.js
  └─ styles.css
        │
        ▼
Python HTTP Server (server.py)
  ├─ 静态资源分发
  ├─ REST API
  ├─ SQLite 读写
  ├─ PDF 解析 / 渲染
  ├─ AI Provider 适配
  └─ 翻译任务调度
        │
        ├─ paper_hub.db
        └─ storage/
            ├─ pdfs/
            ├─ covers/
            ├─ renders/
            ├─ translations/
            └─ digests/
```

### 2.2 技术栈

- 前端：原生 HTML + CSS + JavaScript
- 后端：Python 标准库 `http.server` + `ThreadingHTTPServer`
- 数据库：SQLite
- PDF 处理：`PyMuPDF(fitz)`、`pypdf`
- AI 接入：通过 HTTP 调用多家模型服务，统一走后端适配层

### 2.3 部署方式

- 默认本机启动，端口 `8876`
- 启动方式：
  - `python server.py`
  - 或双击 `start.bat`
- 数据、PDF、封面、翻译缓存都保存在项目目录下

## 4. 数据层设计

### 4.1 SQLite

系统只有一张核心表：`papers`

主要字段包括：

- 基础信息：`title`、`title_zh`、`authors_json`、`year`、`venue`、`doi`、`url`
- 管理属性：`status`、`priority`、`rating`、`favorite`、`collection_name`、`category`
- 内容属性：`abstract`、`notes`
- AI 属性：`ai_keywords_json`、`ai_summary`、`ai_summary_zh`、`ai_model`、`ai_enriched_at`、`ai_confidence`
- 文件属性：`cover_image`、`cover_source`、`pdf_path`、`original_filename`
- 阅读属性：`reader_page`、`reader_total_pages`、`reader_scroll`、`reader_zoom`、`last_read_at`
- 审计属性：`added_at`、`updated_at`

说明：

- 作者、标签、AI 关键词以 JSON 字符串形式存储在 SQLite 中
- 读取时由后端转换成前端直接可用的数组结构
- 后端通过 `ensure_column` 做轻量字段迁移，适合当前单机演进模式

### 4.2 文件存储

- `storage/pdfs/`
  - 保存导入后的 PDF 原文件

- `storage/covers/`
  - 保存论文封面
  - 优先使用 PDF 首页渲染
  - 失败时回退为自动生成 SVG 封面

- `storage/renders/`
  - 保存 PDF 每页渲染图片
  - 供阅读器和视觉翻译使用

- `storage/translations/`
  - 保存逐页翻译缓存
  - 同时记录使用的模型、Provider 和翻译模式

- `storage/digests/`
  - 保存论文精华缓存

### 4.3 Provider 配置

- `provider_config.json`
  - 保存当前选中的 AI Provider
  - 保存各 Provider 的模型、接口地址和本地配置

## 5. 核心模块说明

### 5.1 论文库管理

支持功能：

- 新增论文
- 编辑论文
- 删除论文
- 收藏论文
- 设置阅读状态
- 设置优先级
- 设置评分
- 管理标签、合集、摘要、笔记

前端通过筛选器提供：

- 搜索
- 阅读状态筛选
- 合集筛选
- 收藏筛选
- 标签筛选
- 最近更新 / 评分 / 年份 / 优先级 / 收藏排序

### 5.2 PDF 导入

导入流程：

1. 用户上传 PDF
2. 后端保存文件到 `storage/pdfs/`
3. 后端解析 PDF 元数据和前几页文本
4. 自动推断标题、作者、摘要、标签、分类
5. 自动生成封面
6. 写入数据库
7. 如果 AI 已配置，继续触发 AI 整理

自动提取能力包括：

- 标题推断
- 作者提取
- Abstract 粗抽取
- 标签推断
- 分类推断
- 页数统计
- 封面生成

### 5.3 AI 元数据整理

对单篇论文可执行“AI 整理”，补全和优化：

- 中文标题
- AI 摘要
- 中文摘要
- 标签
- AI 关键词
- 分类
- 合集
- 阅读优先级
- 会议/年份猜测
- 置信度

实现特点：

- 后端统一构造 JSON Schema，要求模型返回结构化结果
- 优先保留已有人工字段，避免 AI 全量覆盖
- 输出重点面向中文阅读场景

### 5.4 论文阅读器

阅读器基于渲染后的 PDF 页面图工作，支持：

- 页间导航
- 当前页识别
- 阅读进度保存
- 上次阅读时间记录
- 原文模式
- 中文精读模式
- 嵌入翻译模式

两种中文阅读方式：

- 中文精读
  - 将页面翻译块整理为连续段落
  - 适合顺畅阅读

- 嵌入翻译
  - 将译文按原页面坐标叠加回页面
  - 适合对照原版版式阅读

### 5.5 页面翻译与全文翻译

单页翻译流程：

1. 抽取当前页文本块和坐标
2. 如果当前 Provider 支持视觉能力，则同时提供页面图片
3. 调用模型返回每个文本块的中文结果
4. 保存到翻译缓存
5. 前端按模式渲染

全文翻译流程：

1. 前端发起后台任务
2. 后端创建任务快照
3. 后端用线程池并发翻译多个页面
4. 前端轮询任务状态和缓存结果
5. 完成后展示进度、失败页和状态

当前任务机制特点：

- 默认最多 3 个并发翻译 worker
- 有缓存复用，已翻译页面不会重复调用
- Provider 或模型切换后，旧缓存会失效

### 5.6 论文精华 Digest

Digest 模块聚焦三段内容：

- Abstract
- Method
- Conclusion

输出内容：

- 原文片段
- 中文整理
- 中文要点列表

两种工作模式：

- AI 已配置：使用模型生成高质量 Digest
- AI 未配置：退化为本地抽取结果，至少给出原文片段和已有摘要

### 5.7 AI Provider 管理

系统内置多种 Provider 预设，统一封装为后端适配层。

支持的能力方向：

- OpenAI Responses
- OpenAI 兼容 Chat Completions
- Anthropic Messages
- Gemini generateContent
- 本地兼容端点（如 Ollama、LM Studio）

前端能力包括：

- 切换 Provider
- 配置 Model
- 配置 API URL
- 保存或清空 API Key
- 查看是否支持视觉翻译

## 6. API 设计

### 6.1 配置相关

- `GET /api/config`
  - 返回运行时配置和前端能力开关

- `GET /api/provider-config`
  - 返回 Provider 配置和预设

- `PUT /api/provider-config`
  - 保存当前 Provider 配置

### 6.2 论文管理

- `GET /api/papers`
  - 获取论文列表

- `POST /api/papers`
  - 新增论文

- `PUT /api/papers/{id}`
  - 更新论文

- `DELETE /api/papers/{id}`
  - 删除论文及关联资源

- `POST /api/papers/import-pdf`
  - 上传并导入 PDF

- `POST /api/papers/{id}/ai-enrich`
  - 执行 AI 整理

### 6.3 阅读器相关

- `GET /api/papers/{id}/reader`
  - 获取阅读器页面数据

- `PUT /api/papers/{id}/reader-state`
  - 保存阅读进度

- `GET /api/papers/{id}/reader-translation?page=n`
  - 获取单页翻译

- `GET /api/papers/{id}/reader-translations`
  - 获取缓存翻译集合

- `GET /api/papers/{id}/reader-translation-job`
  - 获取全文翻译任务状态

- `POST /api/papers/{id}/reader-translation-job`
  - 启动全文翻译任务

### 6.4 Digest 与系统

- `GET /api/papers/{id}/digest`
  - 获取缓存或实时生成的论文精华

- `POST /api/papers/{id}/digest`
  - 强制刷新论文精华

- `POST /api/reset`
  - 用示例数据重置数据库

## 7. 关键业务流程

### 7.1 应用启动

1. 初始化存储目录
2. 初始化数据库和缺失字段
3. 若数据库为空，写入示例论文
4. 启动本地 HTTP 服务
5. 前端加载配置和论文列表

### 7.2 导入论文

1. 上传 PDF
2. 保存原文件
3. 解析 PDF
4. 生成封面和初始元数据
5. 写入数据库
6. 可选执行 AI 整理
7. 前端刷新论文库

### 7.3 阅读论文

1. 请求阅读器接口
2. 后端渲染页面图
3. 前端打开阅读器
4. 滚动或切页时更新当前页
5. 延迟保存阅读进度

### 7.4 生成中文阅读内容

1. 请求单页翻译或全文翻译任务
2. 后端抽取文本块
3. 调用 AI
4. 写入缓存
5. 前端渲染成精读视图或嵌入视图

## 9. 当前架构特点与适用性

### 优点

- 结构简单，启动成本低
- 所有核心能力集中在一个进程内，开发调试快
- 本地文件和数据库可直接查看，便于排障
- 对单用户论文阅读和整理场景足够直接

### 局限

- `server.py` 体量较大，后续继续扩展时维护成本会上升
- 前端状态、渲染、接口调用都集中在 `app.js`，模块边界偏弱
- 当前更适合本地个人使用，不适合直接扩成多人系统
- AI、PDF、缓存、HTTP 路由都在同一个单体文件中，测试和替换成本较高

## 10. 后续可演进方向

建议优先级从高到低如下：

1. 拆分后端模块
   - `db.py`
   - `paper_service.py`
   - `pdf_service.py`
   - `ai_service.py`
   - `provider_service.py`
   - `routes.py`

2. 拆分前端模块
   - 状态管理
   - API 层
   - 列表渲染
   - 阅读器
   - Provider 设置

3. 补充工程化基础
   - `requirements.txt` 或 `pyproject.toml`
   - 基础测试
   - 日志与异常分层

4. 扩展产品能力
   - BibTeX / RIS 支持
   - 批量目录导入
   - 更强的全文检索
   - 云同步或多端数据备份

## 11. 一句话总结

当前项目本质上是一个面向个人论文阅读场景的本地 AI Paper Workbench：前端负责交互，后端集中承载存储、PDF 处理和 AI 编排，整体可用性已经完整，但代码结构仍处于适合快速迭代的单体阶段。
